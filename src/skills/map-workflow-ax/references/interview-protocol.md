# 업무 프로세스 인터뷰 프로토콜

이 문서는 인터뷰를 시작하거나 재개할 때 읽는다. 목표는 사용자의 설명을 특정 회사 양식에 맞추는 것이 아니라, 자유로운 설명을 근거 추적이 가능한 업무 프로세스로 함께 확정하는 것이다.

## 목차

1. 대화 계약
2. 증거 상태
3. 인터뷰 단계
4. 사용자 명령 처리
5. 재개와 충돌
6. 조기 AX 요청
7. 별도 구현 작업

## 1. 대화 계약

- 한 응답에서 질문은 정확히 하나만 한다. 하나의 문장에 여러 질문을 합치지 않는다.
- 답변이 길거나 여러 단계를 포함하면 모두 기록하되, 다음 턴에는 가장 중요한 미확인 항목 하나만 묻는다.
- 매 응답에 짧은 진행 표시, 이번에 기록·변경한 내용, 다음 질문 하나를 포함한다.
- 사용자의 표현을 업무 용어로 정리할 수는 있지만 의미를 바꾸지 않는다.
- 자동화·AI·제품 아이디어는 전체 프로세스가 사용자에게 최종 확정되기 전까지 제안하거나 암시하지 않는다.
- 모르는 정보는 `unknown`으로 남긴다. 그럴듯한 관행이나 도메인 지식으로 빈칸을 채우지 않는다.

권장 응답 형식:

```text
[진행] 범위 ✓ · 기본 흐름 3/5 · 분기 1/? · 전체 확인 ○ · AX 분석 잠금

이번에 반영한 내용
- S03 담당자: 재무운영 담당자
- S03 행동: 신청서와 영수증의 금액·일자를 대조

다음 질문
S03에서 이상이 없으면 누가 무엇을 하나요?

명령: 확정 · 수정 · 이전 · 요약 · 완료 · 재개
```

`?`는 전체 개수를 아직 모른다는 뜻이다. 허위 백분율을 표시하지 않는다.

## 2. 증거 상태

모든 값은 다음 세 상태 중 하나로 관리한다.

| 상태 | 정의 | 저장 원칙 |
|---|---|---|
| 사실 | 사용자가 직접 설명했거나 사용자가 명시적으로 확정한 내용 | 가능한 경우 원문을 `source_quotes`에 보존하고 `inferred: false`로 저장 |
| 추론 | AI가 문맥상 제안했으나 사용자가 아직 확인하지 않은 내용 | 프로세스 사실에 병합하지 말고 제안/대기 상태로 저장; 근거와 신뢰도를 표시 |
| 미확인 | 필요한 정보가 없거나 답을 모르는 내용 | `null`, 빈 배열 또는 `unknowns`로 저장; 숫자·담당자·시스템을 임의 생성하지 않음 |

사용자가 추론을 `확정`하면 새 사용자 확인을 근거로 사실로 승격한다. 변경 전 상태는 변경 로그에 남긴다.

## 3. 인터뷰 단계

### A. 범위 설정

다음을 순서대로 확인하되 이미 답한 내용은 다시 묻지 않는다.

1. 업무 이름과 목적
2. 시작 사건과 종료 조건
3. 범위에 포함·제외할 구간
4. 업무 결과를 받는 내부·외부 고객

범위는 초안으로 요약한 뒤 사용자에게 `확정`을 받는다.

### B. 기본 흐름 발견

시작점부터 시간 순서대로 한 단계씩 진행한다. 각 단계에 필요한 최소 필드는 다음과 같다.

- `id`: 안정적인 단계 ID (`S01`, `S02` 등). 순서가 바뀌어도 기존 ID를 재사용하지 않는다.
- `actor`: 실제 수행 주체 또는 역할
- `action`: 동사로 시작하는 수행 내용
- `inputs`: 이 단계가 사용하는 정보·문서·이벤트
- `outputs`: 다음 단계로 전달되거나 저장되는 결과
- `systems`: 사용하는 시스템·채널·파일
- `next_steps`: 정상 경로의 다음 단계 ID 또는 종료
- `source_quotes`: 해당 내용을 뒷받침하는 사용자 표현
- `status`: 제안 중 또는 `confirmed`

필드 전체를 한꺼번에 묻지 않는다. 사용자의 답에서 추출한 뒤 가장 중요한 빈칸 하나만 질문한다. 단계를 요약하고 사용자가 `확정`했을 때만 `confirmed`로 바꾼다.

### C. 판단·승인·인계 확인

기본 흐름이 연결되면 각 단계에서 다음을 확인한다.

- 판단 기준과 결과별 다음 경로
- 승인자, 승인 대상, 반려 시 되돌아가는 지점
- 사람·팀·외부기관 사이의 인계
- 입력 또는 출력의 보관 위치

승인과 실제 실행 권한을 같은 것으로 추정하지 않는다.

### D. 분기·예외 확인

정상 경로를 먼저 완성한 뒤 예외를 탐색한다. 한 턴에 하나의 분기 또는 예외만 다룬다.

각 분기에는 다음을 기록한다.

- 출발 단계 ID
- 조건 또는 발생 사건
- 처리 주체와 행동
- 다음 단계 또는 종료 결과
- 정상 흐름으로 합류하는 지점
- 빈도·영향·처리시간: 사용자가 아는 경우에만 기록

다음 순서로 질문한다.

1. 가장 자주 발생하는 예외
2. 반려·재작업 경로
3. 시스템 오류·입력 누락·기한 초과
4. 금액·위험·권한에 따른 승인 분기
5. 더 이상 중요한 분기가 없는지 확인

분기마다 목적지가 있어야 하며, 존재하지 않는 단계 ID를 연결하지 않는다. 루프는 허용하되 종료 조건을 확인한다.

### E. 운영 맥락 확인

AX 우선순위에 영향을 주지만 사용자가 모르면 미확인으로 둘 수 있다.

- 빈도와 건수
- 단계별 또는 전체 소요시간
- 대기시간과 병목
- 오류·반려·재작업 비율
- 데이터 형식과 품질
- 시스템 접근·연동 가능성
- 개인정보, 기밀정보, 규제, 감사·보존 요구
- 현업이 느끼는 불편과 개선 목표

수치를 요구할 때 추정치를 허용할 수 있으나 반드시 `사용자 추정`으로 표시한다.

### F. 전체 확인

다음 조건을 점검한다.

- 시작과 종료가 명확하다.
- 모든 기본 단계가 `confirmed`다.
- 정상 경로가 끊기지 않는다.
- 알려진 판단, 승인, 반려, 중요한 예외가 연결되어 있다.
- 사실·추론·미확인이 분리되어 있다.
- 미확인 사항이 별도 목록으로 보인다.

전체 표와 경로 요약을 보여준 뒤 반드시 하나의 명시적 질문으로 최종 확인한다.

> 위 업무 흐름을 AX 분석의 기준본으로 확정할까요?

사용자가 확정하기 전에는 AX 분석 잠금을 해제하지 않는다. 확정 후 프로세스를 수정하면 기존 분석을 폐기 또는 구버전으로 표시하고 전체 확인 단계로 되돌린다.

## 4. 사용자 명령 처리

| 명령 | 동작 |
|---|---|
| `확정` | 현재 제안 항목을 확정한다. 전체 확인 화면에서는 프로세스 전체를 확정한다. 무엇을 확정할지 모호하면 대상 하나만 질문한다. |
| `수정 [내용]` | 대상 필드·단계만 수정하고 변경 로그를 남긴다. 대상이 모호하면 하나의 확인 질문을 한다. |
| `이전` | 직전 인터뷰 초점으로 이동한다. 데이터를 삭제하거나 되돌리지 않는다. 취소가 필요하면 수정 의도를 확인한다. |
| `요약` | 현재 범위, 확정 단계, 분기, 미확인 사항과 진도를 보여준 뒤 다음 질문 하나를 이어간다. |
| `완료` | 현재 단계의 종료 가능성을 검사한다. 빠진 필수 연결이 있으면 가장 중요한 빈칸 하나를 묻고, 충분하면 전체 확인으로 이동한다. 자동으로 전체 확정하지 않는다. |
| `재개` | 저장된 세션을 불러와 마지막 확정 지점과 다음 질문을 보여준다. 여러 세션이 있으면 목록을 보여주고 어느 세션인지 한 번만 묻는다. |
| `구현 작업 열기 AXnn` | 최종 검증된 기회 하나에 대해 구현 인계서를 만들고 Codex 앱의 별도 작업을 연다. 인터뷰 작업에서는 코드를 만들지 않는다. |
| `구현 결과 검토 <RESULT.md>` | 인계서의 최신성, 완료 조건, 테스트, 사람 경계를 다시 검사하고 다음 결정 하나를 묻는다. |

명령과 업무 설명이 한 메시지에 함께 있으면 설명을 먼저 반영하고 명령을 실행한다.

## 5. 재개와 충돌

- 매 사용자 답변 직후 디스크에 저장한다. 다음 질문을 하기 전에 저장 성공을 확인한다.
- 재개 시 채팅 기억보다 저장된 `process.json`과 이벤트 로그를 우선한다.
- 저장된 리비전과 메모리의 리비전이 다르면 덮어쓰지 말고 최신 상태를 다시 읽는다.
- 사용자가 다른 업무를 시작하면 기존 세션을 보존하고 새 세션을 만든다.
- 저장 실패 시 성공한 것처럼 말하지 않는다. 현재 답변 내용을 유지한 채 오류를 짧게 알리고 재시도한다.

## 6. 조기 AX 요청

전체 확정 전 사용자가 자동화 아이디어를 요구하면 다음처럼 처리한다.

1. 현재 설명만으로는 실제 업무가 아닌 가정을 분석할 위험이 있음을 한 문장으로 알린다.
2. AX 제안은 잠근 채 인터뷰를 계속한다.
3. 현재 가장 중요한 질문 하나만 한다.

사용자가 명시적으로 조기 가설을 원해도 확정 분석 파일에는 넣지 않는다. 별도 대화 가설로만 표시하고 사실과 섞지 않는다.

## 7. 별도 구현 작업

구현은 인터뷰와 분석이 끝난 뒤 사용자가 특정 기회를 명시적으로 선택했을 때만 시작한다.

- 메인 작업은 업무 사실, 확정 프로세스, AX 분석과 수용 판단의 기준본이다.
- 구현 작업은 확정본에서 만든 읽기 전용 인계서를 받아 코드·프로토타입·테스트만 다룬다.
- 전체 채팅 기록을 복사하지 않는다. 선택 기회와 관련된 확정 단계, 원문 근거, MVP 범위, 자동화 경계, 통제, 가정과 미확인 사항만 전달한다.
- 인계서에는 프로세스 리비전과 함께 `process.json` 및 `ax-analysis.json` 해시를 고정한다.
- 구현 작업이 업무 사실 오류를 발견하면 단계 ID와 근거를 담은 변경 요청만 반환한다. 기준본을 직접 수정하지 않는다.
- 구현 전후에 인계서를 재검증한다. 기준본 또는 분석 해시가 바뀌면 `needs_rebase`로 처리한다.
- 구현 완료는 병합, 배포, 운영 승인 또는 효과 입증을 의미하지 않는다.
