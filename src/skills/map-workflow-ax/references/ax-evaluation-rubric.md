# AX 기회 평가 기준

이 문서는 사용자가 전체 프로세스를 확정한 뒤에만 읽는다. 모든 제안은 확정된 단계 ID와 원문 근거에 연결한다. 근거가 없으면 가정 또는 미확인 사항으로 표시한다.

## 목차

1. 분석 단위
2. 해결 유형 선택
3. 근거 기반 점수화
4. 신뢰도
5. 전용 앱의 제품화 기능
6. 통제 설계
7. MVP 선택
8. JSON 작성 규칙

## 1. 분석 단위

각 업무 단계와 인접한 단계 묶음을 모두 검토한다. 문제를 먼저 기술한 뒤 해결 유형을 고른다. 기술 이름을 먼저 정하고 업무를 끼워 맞추지 않는다.

각 후보에 다음을 기록한다.

- 문제와 영향을 받는 사용자
- 관련 단계 ID와 근거 문장
- 해결 유형 하나
- 자동화 경계: 시스템이 하는 일, 사람이 하는 일, 멈추는 조건
- 제품화 기능, 연동, 통제, 위험
- 사실과 구분된 가정·미확인 사항
- 가치 가설과 검증 지표
- 신뢰도와 우선순위

## 2. 해결 유형 선택

| 유형 | 선택 조건 | 피해야 할 경우 |
|---|---|---|
| `human-only` | 책임을 위임할 수 없는 최종 판단, 관계·윤리 판단, 매우 드문 고위험 예외, 충분한 통제가 불가능한 행위 | 단순히 구현이 어려운 일을 모두 사람 전용으로 분류하지 않음 |
| `ai-assist` | 비정형 문서 이해, 분류, 요약, 초안, 검색처럼 확률적 처리가 유용하고 사람이 검토·승인할 수 있음 | 정확한 규칙 계산이나 무검토 고위험 실행 |
| `rules-api` | 규칙이 명확하고 입력이 구조화되어 있으며 안정적인 API·이벤트·DB 연동이 가능함 | 자연어 판단이 핵심이거나 연동 수단이 확인되지 않음 |
| `rpa` | 규칙은 명확하지만 API가 없고 기존 UI 조작이 불가피함 | API가 가능한 경우, 화면이 자주 바뀌는 경우, 예외가 많은 경우 |
| `dedicated-app` | 결과 저장, 상태·이력, 역할, 협업, 승인, 재사용이 반복적으로 필요해 단발성 프롬프트보다 제품 경험이 중요함 | 한 번 쓰는 분석, 기존 시스템의 작은 설정으로 해결 가능한 경우 |
| `hybrid` | AI 판단·규칙 처리·연동·사람 승인이 하나의 지속적 흐름에 함께 필요함 | 단일 방식으로 충분한 작은 업무를 과도하게 복잡화하는 경우 |

결정 순서:

1. 사람에게 남겨야 할 책임과 금지선을 정한다.
2. 완전 결정적 작업이면 `rules-api`를 우선 검토한다.
3. API가 없을 때만 `rpa`를 검토하고 취약성을 위험에 기록한다.
4. 비정형·확률적 작업이면 `ai-assist`와 검토 지점을 설계한다.
5. 상태·이력·역할·반복 사용 가치가 크면 `dedicated-app`을 검토한다.
6. 여러 방식이 실제로 필요할 때만 `hybrid`를 선택한다.

## 3. 근거 기반 점수화

다음 네 축을 각각 0~3점으로 평가한다. 확인된 근거가 없으면 0점을 주지 말고 `미확인`으로 둔다. 숫자는 후보 간 정렬을 돕는 내부 도구이며 사실처럼 보고하지 않는다.

### 가치

- 0: 개선 효과가 확인되지 않음
- 1: 작은 편의 또는 제한된 사용자에게 효과
- 2: 반복 시간, 대기, 오류, 재작업 중 하나를 의미 있게 줄일 가능성
- 3: 빈도·규모·병목 근거가 있고 여러 사용자 또는 핵심 KPI에 큰 효과 예상

### 실행 가능성

- 0: 필요한 데이터·접근·규칙이 없거나 현재 구현 불가
- 1: 큰 데이터 정비 또는 불확실한 연동이 필요
- 2: 주요 데이터와 규칙이 있고 제한된 PoC가 가능
- 3: 디지털 입력, 안정적 규칙·연동, 관리 가능한 예외가 확인됨

### 제품화 가치

- 0: 단발성 답변이면 충분
- 1: 저장 또는 재사용 한 가지 정도만 필요
- 2: 상태·이력·검색·협업·승인 중 여러 기능이 반복적으로 필요
- 3: 다수 사용자, 지속적 워크큐, 피드백 데이터, 권한·감사 요구가 함께 존재

### 통제 가능성

- 0: 오류가 비가역적·고위험이며 실효적 통제가 없음
- 1: 위험이 크고 사람의 전수 검토가 필요
- 2: 승인, 임계값, 근거 표시, 예외 큐로 위험을 관리 가능
- 3: 가역적이고 영향이 제한되며 자동 검증·모니터링이 가능

내부 우선순위 지수는 다음과 같이 참고한다.

```text
0.35 × 가치 + 0.25 × 실행 가능성 + 0.20 × 제품화 가치 + 0.20 × 통제 가능성
```

- `now`: 대체로 2.2 이상이며 치명적 미확인 사항이 없고 신뢰도가 중간 이상
- `next`: 대체로 1.5 이상이지만 데이터·연동·통제 검증이 먼저 필요
- `later`: 효과가 작거나 선행조건이 큼
- `do-not-automate`: `human-only`이거나 통제로 감당할 수 없는 위험

점수와 임계값을 기계적으로 적용하지 않는다. `human-only`는 항상 `do-not-automate`로 둔다. 낮은 신뢰도의 후보를 `now`로 두지 않는다.

## 4. 신뢰도

- `high`: 관련 단계, 빈도·영향, 데이터·시스템, 예외, 통제가 대부분 사용자에게 확인됨
- `medium`: 핵심 흐름과 문제는 확인됐으나 일부 정량값 또는 구현 조건이 미확인
- `low`: 가정이 가치 또는 실행 가능성 판단을 좌우함

가정을 많이 써서 신뢰도를 높이지 않는다. 중요한 미확인 사항은 `unknowns`와 최종 보고서에 그대로 노출한다.

## 5. 전용 앱의 제품화 기능

AI가 핵심 결과를 만든 다음 사용자가 무엇을 해야 하는지 묻는다.

> 결과를 저장·수정·검색·공유·승인·재사용·학습하려면 어떤 지속 기능이 필요한가?

근거가 있는 기능만 선택한다.

- 업무함, 상태, 담당자 배정, 우선순위
- 저장, 버전, 변경 이력, 비교
- 검색, 필터, 태그, 일괄 처리
- 검토, 의견, 승인, 반려, 재요청
- 마감 알림, SLA, 에스컬레이션
- 원문 근거 표시와 결과-근거 연결
- 예외 큐, 재시도, 수동 인계, 복구
- 사용자 수정 피드백과 재사용 가능한 지식
- 대시보드, 처리량·시간·품질 KPI
- 역할 기반 권한, 인증, 보존, 내보내기
- 기존 시스템·API·문서 저장소 연동

기능 이름만 나열하지 말고 어떤 단계의 어떤 불편을 해결하는지 설명한다.

## 6. 통제 설계

각 후보에 최소 하나의 구체적인 통제를 둔다. 위험과 통제를 한 쌍으로 연결한다.

- 사람 승인: 금액, 중요도, 신뢰도 임계값 또는 외부 실행 전 승인
- 권한 분리: 생성·검토·승인·실행 역할 분리
- 근거성: 원문 링크, 인용, 적용 규칙·모델 버전 기록
- 입력 검증: 필수값, 형식, 중복, 합계, 교차 확인
- 예외 처리: 자동 중단, 예외 큐, 담당자 할당, 재처리
- 보안·개인정보: 최소 수집, 마스킹, 접근 통제, 보존·삭제
- 감사 추적: 입력, 출력, 수정, 승인, 실행 시각과 주체 기록
- 운영 안정성: 타임아웃, 재시도, 중복 실행 방지, 롤백, 비상 중지
- 품질 모니터링: 샘플 검수, 오류율, 드리프트, 거짓 양성·음성
- 비용 통제: 호출량, 처리 한도, 사용량 알림

`사람이 확인한다`만 쓰지 말고 누가, 언제, 무엇을 근거로, 실패하면 어디로 보내는지 적는다.

## 7. MVP 선택

`now` 후보 중 1~3개를 하나의 얇은 종단 흐름으로 고른다.

- 한 사용자 집단과 한 명확한 문제를 우선한다.
- 실제 입력부터 결과 검토·승인까지 연결한다.
- 처음에는 연동 하나만 포함하고 수동 업로드를 허용할 수 있다.
- 비가역적·고위험 실행은 범위 밖 또는 사람 승인 뒤로 둔다.
- 기존 제품 설정이나 간단한 API로 충분하면 새 앱 개발을 제안하지 않는다.
- `scope_in`, `scope_out`, 사용자 스토리, 사람 승인 지점, 성공 지표를 명시한다.

성공 지표 예:

- 처리시간, 대기시간, 재작업률, 누락률, 오류율
- 사람 검토시간, 자동 처리율, 예외율
- 승인 리드타임, SLA 준수율
- 사용자 채택률, 반복 사용률, 수정률

기준값을 모르면 현재 수치를 만들어내지 않는다. `PoC 시작 전 기준선 측정`을 선행 작업으로 두고, 목표치는 검증 가설로 표현한다.

## 8. JSON 작성 규칙

`references/ax-analysis-schema.json`의 필드만 사용한다.

- `evidence.quote`는 확정 단계의 `source_quotes`, `name`, `action`, 또는 확인된 문제·규칙·예외에서 가져온다.
- 하나의 후보가 참조하는 모든 증거의 단계 ID를 `step_ids`에 포함한다.
- 전용 앱 또는 하이브리드는 `product_features`를 비워 두지 않는다.
- 통제와 검증 지표를 최소 하나씩 둔다.
- 가정과 미확인 사항을 사실 문장에 섞지 않는다.
- 프로세스 리비전과 세션 ID를 확정된 `process.json`과 일치시킨다.
- 스키마에 없는 내부 점수는 JSON에 추가하지 않는다.
